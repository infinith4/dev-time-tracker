# PM向け 要件定義で実施する内容

## 目的
要件定義フェーズでPMが実施すべき活動を明確化し、抜け漏れなく進行管理できる状態にする。

## 実施内容

| フェーズ | 実施項目 | 具体内容 | 主担当 | 完了条件 |
|---------|----------|----------|--------|----------|
| 1. 準備 | キックオフ実施 | 目的、スコープ、体制、進め方、意思決定ルールを合意 | PM | キックオフ議事録が共有済み |
| 1. 準備 | ステークホルダー特定 | 利害関係者、責任範囲、承認者を整理（RACI） | PM | ステークホルダーマップ承認済み |
| 1. 準備 | 計画策定 | 要件定義計画（タスク、日程、マイルストーン、レビュー日）を作成 | PM | 要件定義計画書が承認済み |
| 2. 収集 | 現状把握 | 現行業務、既存システム、課題、制約条件を整理 | PM/BA | 現状課題一覧がレビュー完了 |
| 2. 収集 | ヒアリング/ワークショップ | ユーザー部門・運用部門・開発部門から要求を収集 | PM/BA | ヒアリング結果が記録・共有済み |
| 2. 収集 | 要求整理 | 要求を機能/非機能/運用/制約に分類し重複を統合 | BA | 要求一覧の重複・矛盾が解消済み |
| 3. 定義 | スコープ定義 | In Scope / Out of Scope を明文化 | PM | スコープ境界が関係者合意済み |
| 3. 定義 | 機能要件定義 | ユースケース、ユーザーストーリー、受入基準を定義 | BA/PO | FR/UC/US の初版作成完了 |
| 3. 定義 | 非機能要件定義 | 性能、可用性、セキュリティ、監視、運用要件を定義 | PM/Architect | NFR の初版作成完了 |
| 3. 定義 | 優先度付け | Must / Should / Could / Won't で優先度決定 | PM/PO | 優先度ルールで全要件分類済み |
| 4. 検証 | 実現性確認 | 技術・コスト・スケジュールの観点で実現性レビュー | PM/TL/Architect | 実現性レビュー記録が承認済み |
| 4. 検証 | 矛盾/欠落チェック | 要件間の整合性、依存関係、抜け漏れを点検 | PM/BA | 指摘事項が解消済み |
| 4. 検証 | トレーサビリティ整備 | 要件と設計・テストの対応関係を紐付け | PM/BA | traceability が更新済み |
| 5. 合意 | レビュー会実施 | ステークホルダー合同レビューで変更点を確定 | PM | レビュー議事録と差分が確定 |
| 5. 合意 | 承認取得 | 承認者から正式承認を取得しベースライン化 | PM/Sponsor | 承認記録と版情報が確定 |
| 5. 合意 | 変更管理開始 | 承認後の変更要求フロー（CR）を運用開始 | PM | 変更管理ルールが周知済み |

## 管理指標（PM）
- 計画進捗率（完了タスク数 / 全タスク数）
- レビュー指摘密度（指摘件数 / 要件件数）
- 承認リードタイム（レビュー完了日から承認日まで）
- 変更要求件数（ベースライン後）

## 要件定義の完了条件（Exit Criteria）
- 機能要件・非機能要件・制約条件が承認済みであること。
- 要件情報に基づき、システム規模（画面数、主要機能数、外部連携数、データ量、開発工数の概算）を見積もれる状態であること。
- 設計作業に必要な引継ぎ情報（スコープ、要件優先度、ユースケース、非機能要件、技術制約、未決事項、前提条件）が整理され、設計担当へ共有済みであること。
- 要件ベースライン確定後の変更管理ルール（CRフロー）が有効化されていること。

## 要件定義と概算見積・契約の関係
- 要件定義の完了時点で、システム規模が明確になる。
- 明確化した規模（機能量、画面数、連携数、データ量、非機能要件）を基に、システム構築の概算費用を見積もる。
- この概算費用見積は、次フェーズであるシステム構築契約へ進むための判断材料となる。
- 要件定義書および関連成果物は、概算費用見積の根拠資料として扱う。

## 判定シーン別チェックリスト（事前準備）
もめ事が発生してから判定方法を決めるのではなく、計画時点で以下の判定基準を準備しておく。

### 1. 優先順位付け（複数候補の取捨選択）
- [ ] 候補が一覧化され、比較対象が同じ粒度で定義されている
- [ ] 評価軸（事業価値、利用者影響、実現コスト、リスク、期限）が合意されている
- [ ] 優先度ルール（Must / Should / Could / Won't）が関係者に共有されている
- [ ] 依存関係と前提条件（先行実装が必要な要件）が明示されている
- [ ] 判定結果の理由と却下理由が記録されている

### 2. 作業の開始・終了判定（準備十分性と議論十分性）
- [ ] 開始判定: 目的、対象範囲、入力情報、担当者、期限が明確である
- [ ] 開始判定: 必要な事前資料（現状分析、制約、関連要件）が揃っている
- [ ] 進行判定: 論点、懸念、未決事項が可視化され、担当と期限が設定されている
- [ ] 終了判定: 議論・分析結果が文書化され、レビュー可能な状態である
- [ ] 終了判定: 次工程への引継ぎ情報（決定事項、保留事項、前提条件）が整理されている

### 3. 合意・承認判定（関係者の異議有無）
- [ ] 承認対象（定義書、決定事項、変更点）が明確である
- [ ] 関係者全員が最新版を確認できる状態である
- [ ] レビュー指摘に対する対応結果（反映/却下理由）が記録されている
- [ ] 異議の有無が確認され、未解決論点の扱いが合意されている
- [ ] 最終承認者と承認日時が記録され、ベースライン版が確定している
